This manual explains how to use W.P. Stallman to generate manifests and installer stubs for your WordPress plugins. The workflow has two main phases: creating the manifest and then building the installer.

1. Manifest Tab

The Manifest tab is where you connect to your database, select the SQL objects you want to package, and generate the manifest JSON file.

1.1 Connect to database

  • Enter your connection string in the Connection String box.
    Example:

    server=localhost;uid=root;pwd=;database=wp_stallman_design

  • Specify your DB Prefix (default is wp_).

  • Choose an Installer Class Name. This is the PHP class that will be generated for your installer.

1.2 Select SQL objects and seed data

  • Check or uncheck the boxes next to Tables, Views, Stored Procedures, and Triggers to include them in the manifest.
  • Use Include All / Exclude All buttons to bulk-select.
  • To control seed data:
    • Use the Seed column to decide per table.
    • Use the Seed All / Seed None buttons for bulk control.
  • (See Manifest Tab screenshot for example.)

1.3 Preview your manifest

  • The Manifest Preview panel (bottom of tab) shows the JSON that will be generated.
  • It includes:
    • Database name
    • Installer class
    • Objects included (with schema info)
    • Seed data flags

1.4 Save or copy the file to the clipboard

  • Click Save Manifest… to download a .json file.
  • Or use Copy to Clipboard to copy the manifest directly for pasting elsewhere.

2. Installer Tab

The Installer tab takes a manifest JSON and generates a WordPress installer class plus test stubs.

2.1 Open your JSON file

  • Click Choose Manifest JSON… and select the file generated earlier.
  • Once loaded, you’ll see the included objects listed.

2.2 Set your overrides for the database and class name

  • Enter a DB Prefix Override if you want to change the default wp_.
  • Set a different Installer Class Name if needed.
  • (Defaults are taken from the manifest.)

2.3 Choose output options for seed data and test stub

  • Include test stub and main plugin class – generates a runnable PHP test file.
  • Include seed data globally – toggles whether seed data is included (can still be controlled per table).

2.4 Confirm your tables and seed data

  • Review the Tables list.
  • Verify each table’s inclusion and whether seed data is toggled.

2.5 Confirm your other SQL objects

  • Review the lists of Views, Stored Procedures, and Triggers.
  • Use Include All / Exclude All buttons as needed.

2.6 Generate your preview

  • Click Generate Preview to produce the installer PHP code in the Preview panel.

  • Example stub:

    $installer = new WP_Stallman_Design($wpdb);
$installer->install();
$installer->populate();

2.7 Confirm your output and save a zip with a test stub included

  • Choose the output file type (e.g., w-p-stallman-design-installer.php).
  • Use Create Zip to download a bundle with the installer class and test stub.
  • Or Copy to Clipboard / Save This File… for individual files.

3. Testing your stub

Once you have the installer stub, you should test it against a WordPress environment.

3.1 Upload your test stub to see if any errors occur

  • Place the generated PHP files in your WordPress plugin folder.
  • Run the test stub to perform installation and optional seeding.
  • Watch for any SQL or PHP errors.

3.2 Manually verify the integrity of the database

  • Check phpMyAdmin or the MySQL CLI to confirm:
    • Tables are created with expected schema.
    • Views, stored procedures, and triggers are present.
    • Seed data is correctly populated.

4. Manifest File Structure

The manifest is a JSON file describing everything needed for your installer. Example elements:

4.1 Database details

"database": "wp_stallman_design",
"generatedAt": "2025-09-02T21:44:33.6875369Z",
"defaultPrefix": "wp_",
"installerClass": "WP_Stallman_Design",
"includeSeedData": true

4.2 SQL Objects, etc.

  • Tables – with original name, full name, comments, row limits, and seed data toggle.
  • Views – list of included views.
  • Stored Procedures – list of routines to include.
  • Triggers – list of triggers.
